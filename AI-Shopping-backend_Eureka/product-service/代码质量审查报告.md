# 商品服务代码质量审查报告

审查范围：`product-service/src/main/java`

审查重点：复杂度、耦合、多余代码。

验证背景：`mvn test -pl product-service --also-make` 通过，147 tests，0 failures/errors。

## 高优先级

### 1. `ProductServiceImpl` 职责过重，耦合过高

位置：`src/main/java/com/gzasc/aishopping/product/service/impl/ProductServiceImpl.java:46`

现象：同一个类同时处理商品查询、商家管理、上下架、库存恢复、图片写入、图片 URL 查询、店铺 Feign 调用、店铺缓存、DTO 组装前置数据。

影响：类达到 439 行，修改任一业务都容易影响其他路径；测试 mock 依赖也会越来越重。

建议：拆成 `ProductQueryService`、`SellerProductService`、`ProductCommandService`、`ProductImageService`、`ProductShopInfoService`、`InventoryService` 等。

### 2. `ProductService` 接口过宽

位置：`src/main/java/com/gzasc/aishopping/product/service/ProductService.java:17`

现象：一个接口同时暴露用户端查询、商家端查询、内部服务查询、库存、上下架、图片创建/更新。

影响：调用方依赖大接口，职责边界模糊，实现类自然膨胀。

建议：按场景拆接口：买家查询、商家管理、内部查询、库存、图片命令。

### 3. 用户端、商家端、内部端复用同一个详情查询

位置：

- `src/main/java/com/gzasc/aishopping/product/controller/ProductUserController.java:43`
- `src/main/java/com/gzasc/aishopping/product/controller/ProductSellerController.java:30`
- `src/main/java/com/gzasc/aishopping/product/controller/internal/InternalProductController.java:45`
- `src/main/java/com/gzasc/aishopping/product/service/impl/ProductServiceImpl.java:117`

现象：三个入口都调用 `getProductById`，服务层没有区分用户可见、商家归属、内部查询语义。

影响：可见性和权限规则无法从方法名/服务边界看出，后续容易误用。

建议：拆成 `getBuyerVisibleProductDetail`、`getSellerProductDetail`、`getInternalProductDetail`。

### 4. 可售过滤大量放在 Java 内存中

位置：

- `src/main/java/com/gzasc/aishopping/product/service/impl/ProductServiceImpl.java:129`
- `src/main/java/com/gzasc/aishopping/product/service/impl/ProductServiceImpl.java:164`
- `src/main/java/com/gzasc/aishopping/product/service/impl/ProductServiceImpl.java:229`
- `src/main/java/com/gzasc/aishopping/product/service/impl/ProductServiceImpl.java:238`
- `src/main/java/com/gzasc/aishopping/product/service/impl/ProductServiceImpl.java:345`
- `src/main/java/com/gzasc/aishopping/product/mapper/ProductMapper.java:33`
- `src/main/java/com/gzasc/aishopping/product/mapper/ProductMapper.java:74`

现象：先查数据库，再 `.filter(Product::isSale)`；价格分页先 `LIMIT/OFFSET` 后过滤。

影响：分页语义可能错误，性能也差；某页可能过滤后为空，但后续页仍有可售商品。

建议：把 `is_sale = true` 下推到 SQL，并为可售查询单独命名 mapper 方法。

### 5. 上架状态有双重数据源

位置：

- `src/main/java/com/gzasc/aishopping/product/mapper/ProductMapper.java:39`
- `src/main/java/com/gzasc/aishopping/product/mapper/SalableProductMapper.java:10`
- `src/main/java/com/gzasc/aishopping/product/service/impl/ProductServiceImpl.java:320`

现象：同时维护 `products.is_sale` 和 `salable_products`。

影响：重复上架、事务部分失败、数据修复都会造成不一致风险；`addSalable` 不是幂等插入。

建议：明确单一事实来源；若 `salable_products` 是派生表，应做幂等写入、唯一约束和一致性校验。

## 中优先级

### 6. 库存逻辑分散，Mapper 越界操作

位置：

- `src/main/java/com/gzasc/aishopping/product/service/impl/ProductServiceImpl.java:266`
- `src/main/java/com/gzasc/aishopping/product/service/impl/ProductReservationServiceImpl.java:29`
- `src/main/java/com/gzasc/aishopping/product/mapper/ProductReservationMapper.java:28`

现象：恢复库存放在 `ProductServiceImpl`，预占/确认在 `ProductReservationServiceImpl`，`ProductReservationMapper` 直接更新 `products.stock`。

影响：库存规则散落，后续加冻结库存、库存流水会改多个地方。

建议：抽 `InventoryService` 和 `ProductStockMapper`，统一管理 reserve/confirm/release/restore。

### 7. DTO 和 Converter 中存在大量 `@Deprecated` 但仍被主流程使用

位置：

- `src/main/java/com/gzasc/aishopping/product/dto/ProductWithImageAbstractDTO.java:10`
- `src/main/java/com/gzasc/aishopping/product/dto/ProductWithImageDetailDTO.java:12`
- `src/main/java/com/gzasc/aishopping/product/converter/ProductConverter.java:20`
- `src/main/java/com/gzasc/aishopping/product/service/impl/ProductServiceImpl.java:125`
- `src/main/java/com/gzasc/aishopping/product/service/impl/ProductServiceImpl.java:143`
- `src/main/java/com/gzasc/aishopping/product/service/impl/ProductServiceImpl.java:161`

现象：废弃 DTO/转换方法仍是核心返回模型。

影响：`@Deprecated` 失去信号价值，后续开发者不知道该迁移还是继续使用。

建议：若仍是主模型，移除废弃标记；若真要废弃，先建立新 DTO 并完成迁移。

### 8. 批量查询代码重复严重

位置：`src/main/java/com/gzasc/aishopping/product/service/impl/ProductServiceImpl.java:129` 到 `src/main/java/com/gzasc/aishopping/product/service/impl/ProductServiceImpl.java:362`

现象：多处重复执行：查商品、判空、构建 imageUrlMap、收集 shopIds、batchGetShopInfo、converter 转换。

影响：每加一个查询接口都复制一套模板，增加遗漏和行为不一致风险。

建议：抽通用私有方法，如 `buildImageAndShopContext(products)`，再由不同 converter 消费。

### 9. 图片校验逻辑重复放在 Controller

位置：

- `src/main/java/com/gzasc/aishopping/product/controller/ProductSellerController.java:61`
- `src/main/java/com/gzasc/aishopping/product/controller/ProductSellerController.java:92`

现象：创建和更新重复判断空文件、content-type。

影响：校验策略散在入口层，后续支持 webp/大小限制时需改多处。

建议：抽 `ProductImageValidator` 或放入 `ImageStorageService` 前置校验。

### 10. 图片 URL 拼接和反解析耦合配置细节

位置：

- `src/main/java/com/gzasc/aishopping/product/service/impl/ProductServiceImpl.java:392`
- `src/main/java/com/gzasc/aishopping/product/service/impl/ImageStorageServiceImpl.java:63`
- `src/main/java/com/gzasc/aishopping/product/service/impl/ImageStorageServiceImpl.java:72`

现象：`ProductServiceImpl` 拼 `imageBaseUrl + relativePath`，`ImageStorageServiceImpl` 删除时再用 `imageBaseUrl` 反解析。

影响：URL 规则变化会影响多个类；存储服务和业务服务共同理解路径格式。

建议：让 `ImageStorageService` 统一返回完整 URL 或统一返回相对路径，避免双向拼接。

位置：

- `src/main/java/com/gzasc/aishopping/product/controller/ProductUserController.java:31`
- `src/main/java/com/gzasc/aishopping/product/controller/ProductUserController.java:45`

现象：多个方法重复 `try/catch ProductException / Exception`，拼接 `"查询错误：" + e.getMessage()`。

影响：样板代码多，错误响应策略不集中。

### 13. 批量 ids 字符串解析重复且缺少统一组件

位置：

- `src/main/java/com/gzasc/aishopping/product/controller/ProductSellerController.java:47`

现象：两个 Controller 都手动 `ids.split(",")`。

影响：空值、非法数字、空白处理不一致。

建议：使用 `@RequestParam List<Long>` 或抽公共解析器。

### 14. Converter 中列表转换模式重复

位置：

- `src/main/java/com/gzasc/aishopping/product/converter/ProductConverter.java:69`
- `src/main/java/com/gzasc/aishopping/product/converter/ProductConverter.java:92`

- `src/main/java/com/gzasc/aishopping/product/converter/ProductConverter.java:152`

现象：每种 DTO 都重复取 imageUrl、shopInfo 的模板。

影响：转换逻辑膨胀，新增 DTO 会继续复制。

建议：抽 `resolveImageUrl`、`resolveShopInfo` 或引入上下文对象。

### 15. 测试运行时有定时任务干扰日志

位置：`src/main/java/com/gzasc/aishopping/product/service/impl/ProductReservationServiceImpl.java:100`

现象：测试期间调度任务访问空 H2 表并打印异常栈，但测试仍通过。

影响：噪声大，掩盖真实失败。

建议：测试 profile 关闭 scheduler，或让任务依赖独立开关。

## 结论

主要质量问题集中在三类：

1. 服务接口和实现类过大，职责边界不清。
2. 查询语义不稳定，可售/商家/内部场景混用。
3. 历史遗留 DTO、废弃方法、重复转换/校验代码较多。

如果要优先重构，建议顺序：先拆查询语义和可售 SQL，再拆图片/店铺上下文，最后清理废弃接口。
